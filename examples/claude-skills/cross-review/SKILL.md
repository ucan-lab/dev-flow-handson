name: cross-review
description: |
  現在のフィーチャーブランチの累積差分 (base..HEAD) を、辛口/甘口の 2 ロールで
  並列レビューさせ、両者の指摘をすり合わせて統合レポートを返す。
  単一 AI レビューの視点偏りをクロスバリデーションで補正するためのスキル。
disable-model-invocation: true
allowed-tools: Bash(git *), Read, Grep, Glob, Agent
argument-hint: "[base-branch]"
---

## 起動時コンテキスト

- 現在ブランチ: !`git branch --show-current`
- 作業ツリー状態: !`git status --short`
- 推定マージベース (vs origin/main): !`git merge-base origin/main HEAD 2>/dev/null || echo "n/a"`
- 累積コミット (vs origin/main): !`git log origin/main..HEAD --oneline 2>/dev/null | head -30`
- 累積差分サマリ (vs origin/main): !`git diff origin/main...HEAD --stat 2>/dev/null | tail -30`

ベースブランチは引数 `$ARGUMENTS` の有無で分岐するため、起動時コンテキストは `origin/main` 仮定の参考値として扱う。

## 事前チェック

1. `git branch --show-current` で現在ブランチを取得。`main` / `master` / `develop` 上なら中止。
2. ベースブランチを決定:
   - 引数 `$ARGUMENTS` が指定されていればそれを採用 (例: `/cross-review develop`)。`git rev-parse --verify <base>` で存在確認し、見つからなければユーザーに聞き直す。
   - 引数が無ければ既定 `origin/main`。無ければ `master` / `develop` / ローカル `main` を順に試す。
3. 採用するベースをユーザーに提示し承認を取る (引数で指定された場合は `指定どおり <base> で進めます。` と短く確認)。
4. 累積差分が 0 行なら終了。
5. 累積差分が **2000 行超** のときは、観点を絞るか / コミット単位レビューに切り替えるか / そのまま続行するかを確認。

## レビューフロー

### ステップ 1: 2 ロール並列起動

承認後、**1 メッセージ内で 2 つの Agent を並列に起動** する (Claude Code は同一メッセージ内の複数 Agent 呼び出しを並列実行する)。

両エージェントには **同じ diff 本文** を渡し、ロール定義 (口調・観点・厳しさ) のみ差し替える。

#### 共通プロンプト要素

両ロールに共通で渡す内容:

- ベースブランチと現ブランチ名
- `git diff <base>...HEAD` の本文 (差分が大きい場合はファイル単位に分けて全文)
- 累積コミット一覧 (`git log <base>..HEAD --oneline`)
- 出力フォーマット指定 (下記)
- 指摘には必ず `<path>:<line>` を付けること
- 推測ではなく diff 上で確認できる箇所のみ報告すること

#### 辛口ロール

```text
あなたは辛口のシニアレビュアーです。マージブロックすべき欠陥を見逃さないことが最優先です。

観点:
- 設計の妥当性 (責務分離、抽象化の早すぎ/遅すぎ)
- 命名整合 (既存コードベースとの一貫性、誤解を招く命名)
- エラーハンドリング (例外漏れ、握りつぶし、エラーメッセージのリーク)
- セキュリティ (入力検証、SQLi/XSS、権限チェック、シークレット混入)
- テスト網羅 (新規ロジックに対するテストの有無、境界値、異常系)
- パフォーマンス (N+1、O(n^2) ループ、不要な同期処理)
- 可読性・保守性

スタンス: 改善余地はすべて推奨修正以上で出してください。「動くなら通す」ではなく「ベストか」で判断してください。

出力フォーマット (Markdown、見出しは日本語固定。`Must` / `Should` などの英語ラベルは使わない):

## 要修正
- <path>:<line> - <指摘>

## 推奨修正
- <path>:<line> - <指摘>

## サマリ
<1-2 文>
```

#### 甘口ロール

```text
あなたは甘口のレビュアーです。マージを止めるべき欠陥だけ指摘してください。スタイルや命名、軽微な改善は無視します。

観点 (これ以外は無視):
- バグ (動作が壊れる、データ破損リスク、無限ループ、競合状態)
- セキュリティ事故 (シークレット混入、認証回避、権限昇格、SQLi/XSS)
- データ消失リスク (誤った DELETE / DROP、unsafe な migration)
- ビルド破綻 (構文エラー、未定義参照)

スタンス: 「動いて事故らないなら通す」。命名・スタイル・テスト不足・軽微なリファクタ提案は出さない。

出力フォーマット (Markdown、見出しは日本語固定):

## 要修正
- <path>:<line> - <指摘>

## サマリ
<1-2 文>
```

### ステップ 2: すり合わせ

両エージェントの結果が揃ったら、**指摘を箇所 (path:line もしくは同一トピック) でグルーピング** し、以下のロジックで分類する:

| 辛口     | 甘口     | 分類             | 重大度                                                  |
| -------- | -------- | ---------------- | ------------------------------------------------------- |
| 指摘あり | 指摘あり | **合意指摘**     | 確定 (両者が指摘するのでほぼ確実)                       |
| 指摘なし | 指摘あり | **重大度高**     | 最優先 (甘口がブロッカー認定する箇所は本当に危険)       |
| 指摘あり | 指摘なし | **要すり合わせ** | 採否はユーザー判断 (辛口の改善提案 vs 甘口のスルー判定) |
| 指摘なし | 指摘なし | 出力しない       | -                                                       |

トピックの突合は次の手順で行う:

1. 完全一致の `path:line` で揃っているものを最初にマージ。
2. 同じファイルの近接行 (±3 行以内) かつ指摘内容が概ね同じものは **合意指摘** に寄せる。判断に迷ったら「合意」ではなく「要すり合わせ」へ寄せる。
3. ファイル横断の同一トピック (例: 「全体的にエラーハンドリング不足」など総論) は分類せず **サマリ** に回す。

### ステップ 3: 不一致が多いときのオプション

`要すり合わせ` が **5 件以上** のときは、ユーザーに次の選択肢を提示してよい (任意):

- **3 人目のジャッジを呼ぶ**: もう 1 つ Agent を起動し、辛口/甘口両方の指摘を渡して中立的な判定 (採用 / 却下 / 議論) をさせる
- **そのまま提示する**: ユーザー自身が判断する

ジャッジを呼ぶ場合のロール:

```text
あなたは中立のテックリードです。辛口レビュアーが出した指摘を、甘口レビュアーがスルーしました。
辛口の主張と diff の該当箇所を踏まえ、各指摘について「採用 (要修正/推奨修正)」「却下」「保留 (要議論)」のいずれかを判定し、
1 行で理由を添えてください。
```

## 出力フォーマット (最終レポート)

最終レポートの見出し・ラベルはすべて日本語で出すこと (`Must` / `Should` / `Nits` などの英語ラベルは使わない)。

### 出力原則 (読みやすさのため必ず守る)

- **進行ログを結果に混ぜない**: 「並列起動します」「⏺ N agents finished」などの進捗表示は最終レポートと同じ流れに出さない (内部処理として実行し、結果出力は最終レポートだけにする)。
- **空セクションは出さない**: 0 件の区分は見出しごと省略する。「（なし）」とだけ書いた見出しを残さない。
- **「言及なし」の繰り返し禁止**: セクション見出しで「辛口のみ」と分類済みなので、各項目内に `甘口: 言及なし` のような片側ロール無言及行を書かない。
- **推奨アクション語彙は 3 値に正規化**: `採用推奨` / `仕様確認` / `保留・任意` の 3 段階のみ。「採用寄り」「保留〜採用」「却下」などの中間語彙は使わない (ジャッジ判定セクションは別)。
- **項目内の行数は最大 2 行**: 1 行目に「箇所と指摘要約」、2 行目に必要なら「根拠 / 反論 / 補足」。ロール意見が割れた時のみ 2 行目で並記。

### テンプレート

```markdown
## サマリ

- ベース: <base>
- 範囲: <commits> コミット / <files> ファイル / +<add>/-<del> 行
- 辛口の所感: <1-2 文>
- 甘口の所感: <1-2 文>
- 合意指摘 / 重大度高: <X> 件 / <Y> 件 ※両方 0 のときは「いずれも 0 件」とだけ書く
- 要すり合わせ: <Z> 件 (採用推奨 <a> / 仕様確認 <b> / 保留・任意 <c>)

## 合意指摘 (両ロールが指摘)

- <path>:<line> - <指摘要約>

## 重大度高 (甘口のみ指摘)

- <path>:<line> - <指摘要約>

## 要すり合わせ (辛口のみ指摘)

### 採用推奨

- <path>:<line> - <指摘要約>

### 仕様確認

- <path>:<line> - <指摘要約>
  辛口: <根拠> / 甘口: <反論または静観の理由>

### 保留・任意

- <path>:<line> - <指摘要約>

## (オプション) ジャッジ判定

- <path>:<line> - 採用 / 却下 / 保留: <1 行理由>
```

### 「要すり合わせ」のサブセクション分類ロジック

辛口のみが指摘した項目を、3 サブセクションに振り分ける基準:

| 分類           | 条件                                                                                                                                        |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| **採用推奨**   | 辛口の元出力で `要修正` だった、または明白なバグ・規約違反・デバッグ残置 (`console.log` / 未使用コード)・テスト欠如など反論余地が小さいもの |
| **仕様確認**   | 機能仕様の解釈で結論が変わるもの (例: 「編集時に完了状態をリセットすべきか」)、または辛口/甘口で意見が真っ向から割れたもの                  |
| **保留・任意** | リファクタ提案、抽象化、本 PR スコープ外、UX 改善の任意レベル                                                                               |

迷ったら一段下 (`採用推奨` から `仕様確認`、`仕様確認` から `保留・任意`) に倒す。各サブセクションが 0 件のときはそのサブ見出しごと省略する。

### ゼロ件・全件ゼロの扱い

- 個別セクションが 0 件: 見出しごと省略
- 合意指摘・重大度高が両方 0 件: サマリ行で「いずれも 0 件」とだけ書き、本文側のセクションは出さない
- 要すり合わせのサブセクションが 0 件: そのサブ見出しごと省略
- すべて 0 件: 本文は出さず `両ロールともに指摘なし。` の 1 行のみ

## 禁止事項

- 自動でコード修正・コミット・amend をしない (レビューのみ)。
- 同一メッセージで 2 ロールを並列起動すること。**直列 (1 つずつ) で呼ぶと無駄に時間がかかる**。
- 推測で指摘を増やさない。両エージェントの出力に存在しない指摘を最終レポートに混ぜない。
- ユーザー承認なしにジャッジ (3 人目) を呼ばない。
- `git push` / `git reset` / `git rebase` 等の状態変更コマンドは使わない。

## 最後に伝えること

最終レポートの末尾に **1 文だけ** 次のフォローアップを添える。複数の選択肢を並べない (現在の状態で最有効な 1 つに絞る)。

- 合意指摘 / 重大度高がある場合: 「先に <件数> 件のブロッカー候補を確認してください。」
- 要すり合わせの「採用推奨」が残っている場合: 「採用推奨 <件数> 件から取り込みを検討してください。」
- 不一致が 5 件以上で意見が割れている場合のみ: 「中立ジャッジを呼ぶなら『ジャッジ』と返信してください。」
- 全件 0 件の場合: 「修正不要。マージしてよい状態です。」

`/code-review` 再実行の案内などは原則出さない (ユーザーが必要なら自分で実行する)。
