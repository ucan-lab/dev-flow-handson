# Section 2: 自作 Claude スキルによる、ブランチ差分のレビュー

使うツール: `/code-review` (軽量) と `/cross-review` (クロスバリデーション) の 2 スキル

## Why — なぜやるか

- PR を出す前に AI に diff をセルフレビューさせ、人間レビューに回す前に明らかな問題 (デバッグコード残り、シークレット混入、typo、不要差分) を潰したい
- とはいえレビューには **温度差** がある。コミット直後にざっと見たいときと、PR 直前にじっくり見たいときで、必要な深さが違う
- 単一 AI レビューは **視点が偏る**。同じモデルでも「辛口」「甘口」のロールを与えて 2 回レビューさせれば、見落としを補える (= クロスバリデーション)

## Demo — 完成形

`/code-review` (軽量・1 パス):

1. フィーチャーブランチでコミットした状態で `/code-review` を実行
2. AI が `base..HEAD` の累積 diff を読み、Must / Nits の 2 段階で短く報告
3. 修正 → 追加コミット

`/cross-review` (クロスバリデーション):

1. PR 提出前のブランチで `/cross-review` を実行
2. AI が **辛口レビュアー** と **甘口レビュアー** を **並列に起動** (1 メッセージで 2 つの Agent を呼び出し)
3. 両者の指摘をすり合わせ、**合意 / 重大度高 / 要すり合わせ** の 3 区分で統合レポート
4. 修正 → 追加コミット → PR 作成へ進む

## 準備

Section 1 の続きとして、練習ブランチで複数コミットを積んだ状態を前提にします。デバッグ出力や typo をわざと混ぜておくと、レビューの効きが分かりやすくなります。

```bash
# Section 1 の練習ブランチで作業を続ける
git switch feat/version-constant

# わざと「Must」相当を混ぜる
echo "console.log('debug')" >> version.ts
git add version.ts
git commit -m "chore: add temporary debug log"

# わざと「Nits」相当を混ぜる
echo "// TODO: rmove this comment" >> version.ts  # typo: rmove
git add version.ts
git commit -m "docs: add reminder comment"
```

## How — 手順

### 1. 軽量スキル `/code-review` を登録

Claude Code 起動中に `/code-review` を試して、無ければ以下の内容で `~/.claude/skills/code-review/SKILL.md` を作成 (右上のコピーボタン)。

<<< ../../examples/claude-skills/code-review/SKILL.md [~/.claude/skills/code-review/SKILL.md]

#### フロントマターの読み方

| フィールド                 | 値                        | 意味                                                                   |
| -------------------------- | ------------------------- | ---------------------------------------------------------------------- |
| `name`                     | `code-review`             | `/code-review` として登録されるキー                                    |
| `description`              | (本文参照)                | 1 パス軽量レビューであることを明記                                     |
| `disable-model-invocation` | `true`                    | 会話の流れで AI が勝手に呼ばないように。レビューは人間トリガで行いたい |
| `allowed-tools`            | `Bash(git *), Read, Grep` | git 系の読み取りとコード参照のみ許可。`gh` や Web アクセスは不要       |

`Bash(git *)` はワイルドカードで「`git` で始まる Bash コマンドのみ許可」という指定。`git push` や `git rebase` も入ってしまうので、本文末の **禁止事項** で状態変更コマンドを明示的に止めている。

#### `!` 起動時コンテキストでベースブランチを推定する

```markdown
- 推定マージベース (vs origin/main): !`git merge-base origin/main HEAD 2>/dev/null || echo "n/a"`
- 累積コミット (vs origin/main): !`git log origin/main..HEAD --oneline 2>/dev/null | head -20`
```

- 01-commit.md で扱った `!` プレフィックスをここでも活用。
- AI が Bash ツールを叩く前に **どこからどこまでをレビュー対象にするか** が起動時に決まっているので、最初の往復で「ベースは `origin/main` でよい？」とユーザーに確認できる。
- `2>/dev/null || echo "n/a"` のように **失敗してもスキル自体は止めない** 書き方をしておくと、リモートが無いリポジトリでも起動できる。

### 2. 軽量サンプル運用

練習ブランチ上で:

```
/code-review
```

ベースブランチ (`origin/main` 等) の確認を求められるので承認すると、AI が累積 diff を 1 パスで読み、次のような報告を返します。

```
## サマリ
- ベース: origin/main
- 範囲: 3 コミット / 2 ファイル / +5/-1 行
- 所感: 機能追加に加えてデバッグコードと typo が紛れ込んでいる

## Must (要修正)
- version.ts:2 — console.log('debug') が残っている。コミット前に削除を推奨
- version.ts:3 — コメントアウトされた古いコードが残っている

## Nits (任意)
- version.ts:3 — "rmove" は "remove" の typo
```

Must を直してから再度 `/code-review` を回し、`Must / Nits ともに指摘なし。` が出たら次へ進みます。

### 4. クロスバリデーションサンプル運用

同じ練習ブランチで:

```
/cross-review
```

ベース承認後、AI が **1 メッセージ内で 2 つの Agent を並列起動** します。Claude Code の画面上では、2 つの `Agent` ツール呼び出しが同時に走っている様子が確認できます (片方が辛口、もう片方が甘口)。

両者が返ってきたら、AI は指摘を箇所ごとにグルーピングし、次のような統合レポートを出します。

```
## サマリ
- ベース: origin/main
- 範囲: 3 コミット / 2 ファイル / +5/-1 行
- 辛口の所感: テストが無く、デバッグコードも残っている
- 甘口の所感: ブロッカーは debug log のみ
- 不一致件数: 2 件

## 合意指摘 (両ロールが指摘 = ほぼ確実に修正)
- [Must] version.ts:2 — console.log('debug') が残っている

## 重大度高 (甘口のみ指摘 = ブロッカー濃厚)
- (該当なし)

## 要すり合わせ (辛口のみ指摘 = 採否はユーザー判断)
- version.ts:3 — typo "rmove"
  - 辛口: 識別子・コメントは命名整合の観点でブランチ前に直すべき
  - 甘口: 動作に影響しないので無視
  - 推奨アクション: コメントなので採用 (Nits)
- version.ts — テストコードが付いていない
  - 辛口: 公開する定数なら回帰防止のため簡単な assertion テストが欲しい
  - 甘口: 動くので不要
  - 推奨アクション: 保留 (PR ポリシー次第)
```

#### このスキルで覚えておきたいテクニック

**1. マルチエージェント並列起動**

`Agent` ツールを **同一メッセージ内で複数回呼ぶ** と Claude Code は並列に実行します。直列で呼ぶと 2 倍の時間がかかるので、レビューのように **互いに依存しないタスク** はまとめて投げます。`allowed-tools` に `Agent` を含めるのを忘れずに。

**2. ロールプロンプティングでクロスバリデーション**

同じモデル・同じ diff でも、システム指示 (= ロール) を変えると結論が変わります。`/cross-review` はそれを逆手にとり、辛口/甘口の **2 つの極端なロール** で 2 回レビューさせ、両者の重なりと差分を機械的に分類しています。

- **両者一致** → ほぼ確実に修正案件 (合意指摘)
- **甘口のみ指摘** → 甘口がブロッカー認定する箇所は本当に危険 (重大度高)
- **辛口のみ指摘** → 採否は人間判断 (要すり合わせ)
- **両者スキップ** → 出力しない

**3. すり合わせ (Reconciliation) ロジックを SKILL.md に明文化する**

`/cross-review` の SKILL.md には、上の分類を真理値表として書いてあります。AI は曖昧な「いい感じに統合して」より、**表で示された分岐** のほうが安定してこなせます。プロンプトで決定論的な動きを引き出したいときの定石です。

**4. 不一致が多いときの「3 人目のジャッジ」**

`要すり合わせ` が多い場合は、3 つ目の Agent を中立ロールで起動して採否を裁定させるオプションがあります。デフォルトでは呼ばず、ユーザー承認を取ってから起動する設計にしておくと、コストが暴れません。

## 補足

- **使い分けの目安**: コミット直後の素早い確認は `/code-review`、PR 前のじっくりレビューは `/cross-review`。`/cross-review` で直した後にもう一度ざっと確認したいときは `/code-review` で十分。
- **大きな差分**: 1000-2000 行を超えるブランチは、観点を絞るかコミット単位レビューに切り替えましょう。両スキルとも事前に確認を入れます。
- **ロールを増やす**: `/cross-review` の辛口/甘口 2 ロールに、「セキュリティ専任」「テスト専任」を追加して 3〜4 ロール化する発展版も作れます。すり合わせの真理値表を拡張するのが少し手間ですが、特定領域 (例: 認証コードのレビュー) では効果が大きいです。

## 禁止事項 (スキル側でガード済み)

- 自動コード修正 / 自動コミット / 自動 amend
- ベースブランチ未承認の状態で大量の `git diff` を走らせる
- `git push` / `git reset` / `git rebase` 等の状態変更コマンド
- ユーザー承認なしの 3 人目ジャッジ起動
